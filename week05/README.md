##  핵심 이론 정리 (헷갈리기 쉬운 개념 파헤치기)

웹의 기본 동작 원리 중, 백엔드 개발자로서 코드를 작성할 때 반드시 이해하고 있어야 하는 **상태 유지**와 **네트워크 주소 체계**를 중심으로 정리했습니다.

### 1. HTTP는 기억력이 없다? (Stateless)
* HTTP 프로토콜의 가장 큰 특징은 **무상태성(Stateless)**입니다.
* 클라이언트(브라우저)가 서버에 요청을 보내고 응답을 받으면, **서버는 클라이언트의 상태를 기억하지 않고 연결을 끊어버립니다.**
* 장점: 서버가 상태를 유지할 필요가 없어 서버 확장(Scale-out)에 유리합니다.
* 단점: 사용자가 로그인 상태를 유지하거나, 장바구니에 물건을 담는 등의 '연속적인 작업'을 기본적으로 수행할 수 없습니다.

### 2. 쿠키(Cookie) vs 세션(Session) - 상태를 유지하는 마법
HTTP의 한계(무상태성)를 극복하기 위해 등장한 개념입니다. 둘 다 상태 유지를 위해 쓰이지만, **'정보를 어디에 저장하느냐'**가 가장 큰 차이점입니다.

* **쿠키 (Cookie)**
    * **저장 위치:** 클라이언트 (웹 브라우저)
    * **특징:** 서버가 클라이언트에게 발급해 주는 작은 데이터 조각. 클라이언트가 관리하므로 보안에 매우 취약합니다. (임의 조작 가능)
    * **용도:** '오늘 하루 이 창을 보지 않음' 팝업, 장바구니 내역 등 보안이 크게 중요하지 않은 임시 데이터.
* **세션 (Session)**
    * **저장 위치:** 서버 (메모리 또는 DB)
    * **특징:** 중요한 정보(비밀번호, 개인정보 등)는 서버가 들고 있고, 클라이언트에게는 그 정보를 찾을 수 있는 **'세션 ID(열쇠)'**만 쿠키에 담아 줍니다.
    * **용도:** 로그인 상태 유지, 권한 관리 등 보안이 필수적인 작업.

> **결론:** 세션도 결국 세션 ID를 주고받기 위해 내부적으로는 쿠키를 사용합니다. 백엔드 개발자는 민감한 데이터를 절대 쿠키에 직접 담아 보내면 안 됩니다.

### 3. IP, Port, DNS - 내 Spring Boot 서버는 어떻게 호출될까?
서버와 통신하기 위해 알아야 할 주소 체계입니다. 현실 세계의 주소에 비유하면 헷갈리지 않습니다.

* **IP 주소 (아파트 주소):** 네트워크 상에서 컴퓨터(서버)가 어디에 있는지 식별하는 고유한 번호입니다.
* **Port (아파트 동/호수):** 한 컴퓨터 안에는 여러 개의 프로그램이 동시에 돌아갑니다. 그중 어떤 프로그램과 통신할지 결정하는 출입문 번호입니다.
    * *우리가 Spring Boot를 실행하고 브라우저에서 `localhost:8080`을 입력하는 이유?* 내 컴퓨터(localhost)의 8080번 출입문(Port)에서 Spring 내장 톰캣 서버가 기다리고 있기 때문입니다.
* **DNS (전화번호부):** `192.168.0.1` 같은 숫자로 된 IP 주소를 사람이 외우기 쉬운 `www.naver.com` 같은 도메인 이름으로 변환해 주는 시스템입니다.

##  실습 프로젝트 구조 (Spring Boot API)

본 실습에서는 앞서 배운 서버의 역할을 직접 수행하기 위해 Spring Boot 환경을 구축했습니다.

### 기술 스택
- **Language:** Java 21
- **Framework:** Spring Boot 3.2.5
- **Build Tool:** Gradle 8.7
- **Database:** H2 Database (In-memory)
- **ORM:** Spring Data JPA

### 주요 구현 사항
* **개발 환경 설정:** Spring Initializr를 통해 의존성(Spring Web, Spring Data JPA, H2) 추가 및 폴더 구조 세팅.
* **JPA 연동:** `jakarta.persistence`를 활용하여 `Product` 도메인(Entity)을 설계하고, DB 테이블과 매핑되도록 구성.
* **버전 트러블슈팅:** 기존 제공된 템플릿(Java 11, Spring Boot 2.x)을 현재 로컬 환경에 맞춰 **Java 21 및 Spring Boot 3.2.5 규격(jakarta)으로 마이그레이션**하여 빌드 성공.
